1. Changelog v3.0 (suite)

Recommandations d'architecture

RECOMMANDE Consolidation Variables

Le systeme actuel a ~64 variables pour ~24 couleurs uniques (ratio 2.7:1). Recommandation : consolider vers <1.5:1 en supprimant les variables legacy.

  • Supprimer --easy-* (ancien branding)
  • Utiliser echelles 50-900 au lieu de 1-5
  • Supprimer Mustard (doublon Amber)

ARCHITECTURE Tokens Semantiques

Structure recommandee pour le CSS/Tailwind :

/* Primitives (source) */ --owner-500: #9c5698; /* Semantiques (usage) */ --color-owner-primary: var(--owner-500); --color-owner-hover: var(--owner-600); --color-owner-text: var(--owner-700);

Resume des corrections v3.0

  • Echelles 50-900 : Progression L* monotone decroissante pour les 3 roles
  • Gradient : Les 3 primaires (Owner #9c5698, Resident #e05747, Searcher #ffa000) + bridge #c85570
  • UI Accents : Terracotta #C87060 (+15 deg hue), Dusty Rose #D08090 (+10 sat) - Delta E = 12.8
  • Dark Muted : #8A868E au lieu de #78747C - Ratio 5.2:1 vs 4.5:1
  • Mustard : Supprime (doublon Amber, Delta E = 4.2)

2. Gradient Signature

Le gradient Izzico represente les 3 roles qui completent la plateforme : Owner, Resident, Searcher

Philosophie

Les 3 couleurs primaires des roles sont presentes dans le gradient : Owner #9c5698 (0%), Resident #e05747 (50%), Searcher #ffa000 (100%). Les stops intermediaires assurent une transition harmonieuse.

Code CSS

/* Gradient Signature v3 - avec stop bridge pour transition fluide */ background: linear-gradient(135deg, #9c5698 0%, /* Owner - Mauve */ #c85570 20%, /* Bridge - evite la zone boue */ #d15659 35%, /* Transition rouge */ #e05747 50%, /* Resident - Coral */ #ff7c10 75%, /* Transition orange */ #ffa000 100% /* Searcher - Amber (Primary) */ );

Pourquoi le stop "Bridge" ?

Sur le cercle chromatique, la transition directe Mauve (300 deg) vers Rouge (15 deg) passe par des teintes desaturees ("zone boue"). Le stop #c85570 (rose-rouge a 345 deg) maintient la saturation pendant la transition, creant un effet visuel plus harmonieux.

6. Methodologie des echelles

Progression L* CIELAB pour la coherence perceptuelle

Les echelles 50-900 suivent une progression de luminosite CIELAB (L*) monotone decroissante, garantissant une coherence perceptuelle a travers tous les roles. Cette methode assure que les palettes se comportent de maniere predictible et harmonieuse.

Progression L* cible

/* Progression L* cible (approximative) */ 50: L* = 95-97 /* Fonds tres legers */ 100: L* = 91-94 /* Hover fonds */ 200: L* = 80-89 /* Bordures legeres */ 300: L* = 67-84 /* Accents desatures */ 400: L* = 55-78 /* Elements secondaires */ 500: L* = 45-73 /* PRIMARY - Base de la marque */ 600: L* = 38-62 /* Hover boutons */ 700: L* = 30-48 /* Texte accessible (WCAG AA) */ 800: L* = 22-36 /* Texte emphase */ 900: L* = 14-24 /* Maximum contraste */

Note sur le Searcher

Le Searcher (gold/jaune) a naturellement des valeurs L* plus elevees car c'est une couleur intrinsiquement lumineuse. Ceci est normal et attendu - les couleurs jaunes ne peuvent pas atteindre des L* aussi bas que les violets ou oranges sans devenir brunes.

Pourquoi Zinc ?

L'echelle Zinc de Tailwind a ete choisie pour son caractere neutre avec une tres legere sous-teinte froide, qui s'harmonise bien avec les couleurs chaudes du branding Izzico (mauve, coral, gold). Elle evite les gris trop chauds ou trop froids qui pourraient creer des dissonances visuelles.

11. Guide d'Implementation

Instructions pour appliquer le systeme colorimetrique v3 au code

1. Migration des echelles

Remplacer les anciennes variables 1-5 par les nouvelles 50-900 :

/* AVANT */ --owner-1: #9c5698; /* Primary */ --owner-5: #c2566b; /* Light */ /* APRES */ --owner-500: #9c5698; /* Primary */ --owner-50: #F8F0F7; /* Light */ --owner-700: #633668; /* Text */

2. Tailwind Config

Mettre a jour tailwind.config.ts :

colors: { owner: { 50: 'var(--owner-50)', // ... 100-400 500: 'var(--owner-500)', // ... 600-900 DEFAULT: 'var(--owner-500)', text: 'var(--owner-700)', }, }

3. Gradient Signature

Utiliser le nouveau gradient avec stop bridge :

.gradient-signature { background: linear-gradient( 135deg, #9c5698 0%, #c85570 20%, /* Bridge */ #d15659 35%, #e05747 50%, #ff7c10 75%, #ffa000 100% /* Searcher Primary */ ); }

4. Variables a supprimer

Nettoyer les variables legacy :

  • --easy-* (ancien branding)
  • --owner-1 a --owner-5
  • --mustard (doublon amber)
  • Variables dupliquees non semantiques

5. Dark Mode Muted

Mettre a jour la variable dark-muted :

/* globals.css */ .dark { /* AVANT */ --text-muted: #78747C; /* APRES */ --text-muted: #8A868E; }

6. UI Accents

Mettre a jour Terracotta et Dusty Rose :

/* AVANT */ --terracotta: #D08070; --dusty-rose: #D08080; /* APRES */ --terracotta: #C87060; --dusty-rose: #D08090;

11. Guide d'Implementation (suite)

Fichiers concernes et workflow

Fichiers a modifier

Fichier Modifications
app/globals.css Variables CSS principales, echelles 50-900, dark mode
tailwind.config.ts Configuration des couleurs Tailwind
components/*.tsx Mise a jour des classes de couleur
app/**/*.tsx Pages utilisant les anciennes variables

Workflow de migration recommande

  1. Ajouter les nouvelles variables dans globals.css (sans supprimer les anciennes)
  2. Mettre a jour tailwind.config.ts avec les nouvelles echelles
  3. Migrer les composants un par un vers les nouvelles classes
  4. Tester visuellement chaque composant migre
  5. Une fois tous les composants migres, supprimer les anciennes variables
  6. Effectuer un audit WCAG final

Attention

Ne supprimez pas les anciennes variables avant d'avoir migre tous les composants. Gardez les deux systemes en parallele pendant la migration pour eviter les regressions visuelles.
